# تقرير تقني — تطبيقات سلطان للمواد الغذائية
### (sultanoo + مندوب سلطان) — لغرض الربط مع نظام الحسابات

**تاريخ التقرير:** يوليو 2026
**الغرض:** توفير كل المعلومات التقنية اللازمة لمطور نظام الحسابات عشان يقدر يربط البيانات بين النظامين والتطبيقين.

## 1) نظرة عامة على المنظومة

سلطان للمواد الغذائية بيشغّل حاليًا **تطبيقين منفصلين تمامًا**، كل واحد بمعمارية وطريقة تخزين بيانات مختلفة جذريًا. الفرق ده **أهم حاجة** المطور لازم يفهمها قبل ما يبدأ أي ربط:

| | **sultanoo** | **مندوب سلطان** |
|---|---|---|
| **الوظيفة** | تطبيق طلبات للعملاء (B2B ordering) | تطبيق إدارة مندوبين ميداني |
| **نوع التخزين** | مركزي (Google Sheets) — كل الأجهزة بتشوف نفس البيانات | محلي (localStorage) — كل تليفون له نسخته الخاصة بس |
| **فيه سيرفر/باك إند؟** | ✅ أيوه — Google Apps Script | ❌ لأ — التطبيق شغال بالكامل جوه المتصفح |
| **إمكانية الربط الحالية** | متاحة نسبيًا (فيه API جاهز) | **محتاجة شغل إضافي** (مفيش نقطة اتصال خارجية أصلاً) |
| **الاستضافة** | `mahmoudsultan908.github.io/sultanoo/` | `mahmoudsultan908.github.io/mandob-sultan/` |

### 2.6 شكل بيانات الطلب (Order)

```json
{
  "id": "...",

  "total_amount": 0,
  "status": "new",
  "notes": "...",
```

🟢 نفس الملاحظة: فيه عمود **`erp_order_id`** جاهز لنفس الغرض — ربط كل طلب في sultanoo بمعرّفه في نظام الحسابات.

**حالات الطلب (status) الممكنة:**
`new` (طلب جديد) → `reviewing` (قيد المراجعة) → `preparing` (جاري التحضير) → `delivering` (خرج للتسليم) → `delivered` (تم التسليم) / `cancelled` (ملغي)

### 2.7 الـ API الحالي (Actions متاحة عن طريق GET)

استدعاء بالشكل: `{APPS_SCRIPT_URL}?action=ACTION_NAME&...params`

| Action | الوظيفة |
|---|---|
| `getProducts` | كل المنتجات |
| `getCategories` | كل الأقسام |
| `getAreas` | كل المناطق |
| `getOrders&customer_id=X` | طلبات عميل معين |
| `getCustomerByPhone&phone=X` | بحث عن عميل برقم تليفونه |
| `getSettings` | الإعدادات العامة |
| `getBanners` | البانرات |

### 2.8 الإرسال (POST) — لما التطبيق يبعت بيانات جديدة للسيرفر

كل الطلبات POST بتتبعت لنفس الـ Apps Script URL، بجسم JSON فيه حقل `type` بيحدد نوع العملية:

**إنشاء طلب جديد** (`type: "order"`):
```json
{
  "type": "order",
  "id": "...",
  "customer_id": "...",
  "customer_name": "...",
  "customer_phone": "...",
  "total_amount": 0,
  "status": "new",
  "notes": "...",
  "created_at": "...",
  "items": [ "بنود الطلب" ]
}
```

**تسجيل عميل جديد** (`type: "customer"`):
```json
{ "type": "customer", "id": "...", "name": "...", "shop_name": "...", "phone": "...", "area_id": "...", "area_name": "...", "registered_at": "..." }
```

**تحديث بيانات عميل** (`type: "customer_update"`) — نفس حقول التسجيل.

### 2.9 التوصية للربط
بما إن أعمدة `erp_product_id` و `erp_order_id` موجودة بالفعل جاهزة، أسهل طريقة ربط:
1. المطور يضيف Action جديدة في نفس الـ Apps Script (مثلاً `syncToERP`) بتتشغل تلقائيًا لما يتغير status الطلب لـ `delivered`، وتبعت بيانات الفاتورة لنظام الحسابات عن طريق الـ API بتاعه
2. أو العكس: نظام الحسابات يعمل "سحب" دوري (polling) لشيت `Orders` عن طريق `getOrders`، ويملى `erp_order_id` بعد ما يسجل الفاتورة عنده

## 3) تطبيق مندوب سلطان

### 3.1 المعمارية — الفرق الجوهري
⚠️ **التطبيق ده مفيهوش أي سيرفر أو قاعدة بيانات مركزية خالص.** كل حاجة (المبيعات، العملاء، المحلات، الديون، المخزون) متخزنة **جوه متصفح تليفون المندوب نفسه بس** (`localStorage`). يعني:
- لو المندوب مسح بيانات المتصفح، البيانات بتضيع
- تليفون المندوب مش شايف بيانات تليفون مندوب تاني (لو فيه أكتر من مندوب)
- **مفيش نقطة اتصال (API) خارجية أصلاً يقدر نظام الحسابات يتكلم معاها مباشرة**

### 3.2 هيكل البيانات المحلية (كلها مفاتيح localStorage منفصلة)

| المفتاح | المحتوى |
|---|---|
| `days` | كائن (object) مفتاحه تاريخ اليوم `YYYY-MM-DD`، وقيمته كل حركات اليوم ده (مبيعات، تحصيل، مصروفات، توريد، مرتجعات، زيارات) |
| `customers` | كل العملاء |
| `stores` | كل المحلات — **نفس الـ id بتاع الكستمر المرتبط بيه** (العميل والمحل كيان واحد بمعرّف مشترك) |
| `offers` | عروض المنتجات |
| `invOffers` | عروض الفاتورة (خصم % أو خصم نقدي ثابت) |
| `warehouse` | مخزون المخزن الرئيسي |
| `van_stock` | مخزون عربية المندوب |
| `orders` | الطلبات (لو فيه) |
| `audit` | سجل مراجعة لكل العمليات الحساسة (حذف، تعديل صلاحيات) |
| `goal` | هدف المبيعات الشهري |
| `route` | خط السير المثبّت لكل يوم في الأسبوع |
| `cfg` | إعدادات عامة: `{name, route, pin, adminPin, tgToken, tgChatId, lockDays, editMins, visitsTarget}` |

### 3.3 شكل بيانات العميل/المحل

```json
// customers[]
{ "id": "...", "name": "...", "phone": "...", "debt": 0, "debtLocked": false, "creditLimit": 0 }

// stores[]  (نفس الـ id بتاع customers)
{ "id": "...", "name": "...", "phone": "...", "address": "...", "note": "...", "oldDebt": 0, "visits": [] }
```

### 3.4 شكل بيانات حركة بيع يومية (جوه `days[التاريخ].sales[]`)

```json
{
  "id": "...",
  "prodId": "...",
  "name": "...",
  "qty": 0,
  "unit": 0,
  "total": 0,
  "custName": "...",
  "custId": "...",
  "debt": false,
  "ts": 0,
  "invoiceId": "...",
  "void": false

```

فيه أيضًا `collections[]` (تحصيل ديون)، `expenses[]` (مصروفات)، `deposits[]` (توريد فلوس للمخزن)، `returns[]` (مرتجعات) — كل واحدة جوه نفس كائن اليوم بنفس المنطق.

### 3.5 الطريقة الوحيدة الحالية لإخراج البيانات
**تصدير يدوي (Export)** — المندوب بيدوس زرار من داخل التطبيق، بينزّل ملف JSON فيه نسخة كاملة من كل البيانات المذكورة فوق:

```json
{
  "_v": 5,
  "exportDate": "...",
  "CFG": {}, "DAYS": {}, "CUSTOMERS": [], "STORES": [],
  "OFFERS": [], "INV_OFFERS": [], "WAREHOUSE": [], "VAN_STOCK": [],
  "ORDERS": [], "AUDIT": [], "GOAL": {}
}
```
الملف ده بينزل على التليفون، وكمان بيتبعت تلقائي كملف على بوت تليجرام مربوط بالتطبيق (نفس اللحظة).

### 3.6 تكامل تليجرام (اتجاه واحد بس — إشعارات، مش مزامنة)
التطبيق بيبعت رسائل تلقائية على بوت تليجرام (فاتورة بيع جديدة، تحديث حالة زيارة محل، نسخة الباك أب) لكن ده **إشعار بس، مش قناة بيانات ثنائية الاتجاه**. مفيش استقبال أو رد من تليجرام للتطبيق.

### 3.7 خيارات الربط الممكنة لمندوب سلطان (المطور محتاج ياخد قرار هنا)

بما إن التطبيق مفيهوش سيرفر أصلاً، فيه 3 مسارات ممكنة، من الأسهل للأصعب:

**أ) الأسهل (بدون تعديل كود):** نظام الحسابات ياخد ملفات الـ Export (JSON) دوريًا — يدويًا أو عن طريق قراءتها من تليجرام (البوت بيستقبلها تلقائي كل مرة) — ويعمل عليها استيراد (import) بشكل دوري (يومي مثلًا).

**ب) متوسط:** أنا (اللي بنيت التطبيق) أضيف كود جديد فيه يبعت كل عملية بيع/تحصيل بشكل مباشر (real-time) لـ API خاص بنظام الحسابات عن طريق `fetch()`، بمجرد ما المندوب يقفل الفاتورة — يحتاج المطور يجهز endpoint (زي POST `/api/sync-sale`) يستقبل البيانات بنفس شكل الـ JSON فوق.

**ج) الأشمل والأنضف تقنيًا (على المدى الطويل):** نقل التطبيق بالكامل من `localStorage` لقاعدة بيانات حقيقية سحابية (Firebase / Supabase) — بعدها الربط بيبقى مباشر ومباشر real-time من غير أي وسيط. ده تغيير في البنية مش تعديل بسيط، وتقريبًا نفس التوصية اللي طلعت في تقرير تدقيق Sultan ERP سابقًا.

## 4) ملخص سريع للمطور

- **sultanoo عنده باك إند حقيقي (Google Sheets + Apps Script)** — الربط ممكن دلوقتي عن طريق الـ API الموجود، وفيه أعمدة `erp_product_id` / `erp_order_id` جاهزة مخصصة للربط
- **مندوب سلطان مفهوش باك إند خالص** — البيانات كلها على تليفون المندوب بس، والربط محتاج قرار من الحلول التلاتة فوق (أ/ب/ج) قبل ما يبدأ أي شغل فعلي
- لو المطور احتاج أي توضيح تقني إضافي أو تفاصيل أعمق في أي جزء من الكود، ينسّق مع محدّث النظام (Claude) عن طريقي مباشرة
